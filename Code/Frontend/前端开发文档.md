# 校园二手交易平台前端开发文档

> 项目名称：Campus-Secondhand-Trading-Platform  
> 前端技术栈：Vue 3 + TypeScript + Vite + Element Plus  
> 配套技术：Vue Router + Pinia + Axios  
> 文档版本：v1.0  
> 更新日期：2026-07-14  

---

## 1. 文档目的

本文档用于说明校园二手交易平台前端项目的整体结构、目录职责、技术选型、接口组织方式、状态管理方案、路由权限设计、样式规范及推荐开发顺序。

本项目同时包含：

- 普通用户端
- 管理员端
- 登录与注册
- 商品浏览、发布、编辑和管理
- 用户中心
- 收藏
- 消息
- 订单
- 举报与申诉
- 管理员审核和数据管理

因此，前端不应将所有页面和组件简单堆放在 `src/components` 和 `src/views` 中，而应采用：

> 按技术职责分层，同时按照业务模块划分页面和组件。

## 5. 核心目录职责

### 5.1 `views`：路由页面

`views` 中放置能够通过路由直接访问的完整页面。

示例：

```text
/product/12       → ProductDetailView.vue
/product/publish  → ProductPublishView.vue
/admin/products   → ProductManageView.vue
```

页面组件主要负责：

- 获取路由参数
- 调用接口
- 管理页面级数据
- 组织多个业务组件
- 控制加载、空数据和错误状态
- 处理页面跳转
- 处理页面级操作

判断标准：

> 能被路由直接打开的是页面，供页面复用的是组件。

---

### 5.2 `components`：可复用组件

`components` 中放置被多个页面复用的组件。

例如：首页、商品列表和收藏页面都会展示商品卡片，因此应统一封装：

```text
src/components/product/ProductCard.vue
```

不应在多个页面重复编写商品卡片结构。

推荐分类：

| 目录 | 说明 |
|---|---|
| `components/common` | 全项目通用组件 |
| `components/layout` | 页面头部、页脚、侧边栏 |
| `components/product` | 商品业务组件 |
| `components/order` | 订单业务组件 |
| `components/message` | 消息业务组件 |
| `components/admin` | 管理端专用组件 |

---

### 5.3 `api`：后端接口

不要在页面组件中直接写完整接口地址，例如：

```ts
axios.get('http://localhost:5141/api/product')
```

应将请求封装为有业务含义的函数：

```ts
getAvailableProducts()
getProductDetail(productId)
createProduct(data)
updateProduct(productId, data)
```

接口按照业务模块分别放在：

```text
src/api/modules/
```

示例：

```ts
// src/api/modules/product.ts
import request from '@/api/http'
import type {
  Product,
  CreateProductRequest,
  UpdateProductRequest
} from '@/types/api/product'

export function getAvailableProducts() {
  return request.get<Product[]>('/api/product/available')
}

export function getProductDetail(productId: number) {
  return request.get<Product>(`/api/product/${productId}`)
}

export function createProduct(data: CreateProductRequest) {
  return request.post<Product>('/api/product', data)
}

export function updateProduct(
  productId: number,
  data: UpdateProductRequest
) {
  return request.put<Product>(`/api/product/${productId}`, data)
}
```

这样可以避免页面依赖具体 URL。当后端接口地址发生变化时，只需要修改 `api/modules` 中的文件。

---

### 5.4 `types`：TypeScript 类型

接口数据类型不应零散地定义在不同页面中。

例如，商品类型统一放在：

```text
src/types/api/product.ts
```

示例：

```ts
export type ProductStatus =
  | 'available'
  | 'sold'
  | 'removed'

export interface ProductImage {
  imgId: number
  imgUrl: string
  imgIndex: number
}

export interface Product {
  productId: number
  name: string
  price: number
  info: string | null
  status: ProductStatus
  releaseDate: string
  userId: number
  categoryId: number
  categoryName: string | null
  images: ProductImage[]
}

export interface CreateProductRequest {
  name: string
  price: number
  info?: string
  userId: number
  categoryId: number
}

export interface UpdateProductRequest {
  name?: string
  price?: number
  info?: string
  status?: ProductStatus
}
```

类型文件的主要作用：

- 约束接口请求参数
- 约束接口响应结果
- 提供编辑器代码提示
- 减少字段拼写错误
- 便于前后端接口联调

---

### 5.5 `stores`：Pinia 状态管理

Pinia 用于保存多个页面共同需要的数据。

推荐状态模块：

#### `stores/auth.ts`

保存：

- JWT Token
- 当前用户 ID
- 用户名
- 用户类型
- 是否已登录
- 当前用户信息

#### `stores/productDraft.ts`

保存：

- 商品发布草稿
- 商品图片
- 当前填写步骤
- 草稿更新时间
- 是否存在未保存修改

#### `stores/message.ts`

保存：

- 未读消息数量
- 当前会话
- 会话列表状态

#### `stores/app.ts`

保存：

- 侧边栏展开状态
- 移动端导航状态
- 全局加载状态
- 全局配置

并非所有数据都需要放入 Pinia。

只在单个页面使用的数据，例如某个商品详情，可以直接保存在页面的 `ref` 中。

---

### 5.6 `composables`：组合式函数

`composables` 用于封装可重复使用的 Vue 逻辑。

示例：

| 文件 | 用途 |
|---|---|
| `useAuth.ts` | 登录状态和权限判断 |
| `usePagination.ts` | 分页逻辑 |
| `useProductQuery.ts` | 商品查询条件 |
| `useUnsavedChanges.ts` | 离开页面前未保存提醒 |
| `useResponsive.ts` | 响应式布局判断 |

当多个页面出现相同的响应式逻辑时，应优先考虑提取为 composable，而不是复制代码。

---

### 5.7 `utils`：工具函数

`utils` 中存放与 Vue 组件无直接关系的纯函数。

示例：

- Token 存取
- 日期格式化
- 金额格式化
- 表单校验
- 图片地址处理
- 错误信息提取
- LocalStorage 封装

## 6. Axios 请求封装

建议在 `src/api/http.ts` 中创建统一的 Axios 实例。

主要职责：

- 设置后端基础地址
- 设置请求超时时间
- 自动携带 JWT Token
- 统一处理 401
- 统一处理接口错误
- 统一返回响应数据

示例：

```ts
import axios from 'axios'
import { ElMessage } from 'element-plus'

const request = axios.create({
  baseURL: import.meta.env.VITE_API_BASE_URL,
  timeout: 10000
})

request.interceptors.request.use((config) => {
  const token = localStorage.getItem('access_token')

  if (token) {
    config.headers.Authorization = `Bearer ${token}`
  }

  return config
})

request.interceptors.response.use(
  (response) => response.data,
  (error) => {
    if (error.response?.status === 401) {
      localStorage.removeItem('access_token')
    }

    const message =
      error.response?.data?.message ?? '请求失败，请稍后重试'

    ElMessage.error(message)
    return Promise.reject(error)
  }
)

export default request
```

后续可以继续增加：

- 请求取消
- 重复请求控制
- 文件上传
- 下载文件
- Token 过期跳转登录页
- 统一错误码映射

## 9. 路由设计

路由建议按业务拆分：

```text
router/
├─ modules/
│  ├─ public.ts
│  ├─ user.ts
│  └─ admin.ts
├─ guards.ts
└─ index.ts
```

### 9.1 公开路由

```text
/
/login
/register
/products
/products/:productId
```

### 9.2 登录用户路由

```text
/publish
/messages
/orders
/user
/user/products
/user/favorites
```

### 9.3 管理员路由

```text
/admin
/admin/products/review
/admin/products
/admin/users
/admin/orders
/admin/reports
```

### 9.4 路由权限信息

使用 `meta` 标记：

```ts
{
  path: '/admin',
  component: () => import('@/layouts/AdminLayout.vue'),
  meta: {
    requiresAuth: true,
    roles: ['admin']
  }
}
```

建议在 `router/guards.ts` 中统一处理：

- 是否需要登录
- Token 是否存在
- 用户角色是否满足要求
- 无权限时跳转到 403 页面
- 未登录时跳转到登录页
- 登录成功后返回原页面

## 11. UI 状态设计要求

所有页面应考虑以下状态：

- 加载中
- 空数据
- 网络错误
- 无权限
- 表单错误
- 操作成功
- 危险操作二次确认
- 已售或下架
- 用户被封禁
- 数据不存在

推荐组件：

```text
LoadingState.vue
EmptyState.vue
ErrorState.vue
StatusTag.vue
ConfirmDialog.vue
```

推荐处理方式：

| 状态 | 建议 |
|---|---|
| 加载中 | 骨架屏优先 |
| 空数据 | 显示说明和下一步按钮 |
| 网络错误 | 显示重试按钮 |
| 无权限 | 说明原因并提供返回入口 |
| 表单错误 | 显示在字段下方 |
| 操作成功 | 轻量提示并刷新当前数据 |
| 危险操作 | 使用二次确认 |
| 不可操作状态 | 禁用按钮并说明原因 |

## 12. 前后端接口注意事项

当前接口说明文档和 Swagger 中存在部分差异。

### 12.1 路径差异

接口说明文档中可能使用：

```text
/api/auth/register
/api/auth/login
/api/product
/api/users/{userId}
```

Swagger 中可能使用：

```text
/api/User/register
/api/User/{id}
/api/Product
/api/Product/{id}
```

因此，前端页面不应直接依赖接口路径，所有路径都应集中放在 `api/modules` 中。

### 12.2 需要尽快确认的问题

前后端需要统一：

1. URL 是否统一使用小写。
2. 注册接口最终路径。
3. 登录接口最终路径。
4. 用户详情接口最终路径。
5. 发布商品时是否需要前端传 `userId`。
6. 是否由后端从 JWT Token 中获取用户 ID。
7. 删除商品返回 `200` 还是 `204`。
8. 错误响应是否统一为：

```json
{
  "code": "ERROR_CODE",
  "message": "错误说明"
}
```

9. 商品图片的完整访问地址如何拼接。
10. 时间字段是否统一使用 ISO 8601 格式。
11. 商品状态最终使用字符串还是数字枚举。

## 16. 开发规范建议

### 16.1 文件命名

Vue 组件使用 PascalCase：

```text
ProductCard.vue
ProductDetailView.vue
AdminSidebar.vue
```

普通 TypeScript 文件使用 camelCase：

```text
product.ts
usePagination.ts
validators.ts
```

### 16.2 页面命名

路由页面统一以 `View` 结尾：

```text
LoginView.vue
HomeView.vue
ProductListView.vue
```

### 16.3 组件命名

组件名称应表达职责，避免：

```text
Card.vue
List.vue
Form.vue
```

推荐：

```text
ProductCard.vue
ConversationList.vue
ProductForm.vue
```

### 16.4 接口函数命名

推荐：

```ts
getProducts()
getProductDetail()
createProduct()
updateProduct()
deleteProduct()
```

避免：

```ts
request1()
getData()
handleApi()
```

### 16.5 状态变量命名

推荐：

```ts
const loading = ref(false)
const errorMessage = ref('')
const productList = ref<Product[]>([])
```

### 16.6 不建议的做法

- 在页面中硬编码后端地址。
- 在多个页面重复定义同一个 TypeScript 接口。
- 将所有组件放在一个目录中。
- 将所有路由写在一个超长文件中。
- 将所有状态都放入 Pinia。
- 在 `App.vue` 中编写完整页面布局。
- 多个页面复制同一段请求逻辑。
- 只使用 Element Plus 默认蓝色主题。
- 忽略加载、错误和空数据状态。

## 18. 总结

本项目推荐采用以下架构思路：

> 页面按业务模块拆分，组件按复用范围拆分，接口集中管理，类型集中定义，布局彼此独立，全局状态按需使用。

重点原则：

1. `views` 负责完整页面。
2. `components` 负责复用组件。
3. `api` 负责接口请求。
4. `types` 负责 TypeScript 类型。
5. `stores` 负责跨页面共享状态。
6. `layouts` 负责普通用户端、个人中心和管理员端布局。
7. `router` 负责页面导航和权限控制。
8. `assets/styles` 负责平台视觉规范和 Element Plus 主题。
9. 接口路径和后端地址禁止散落在页面中。
10. 目录结构应逐步扩展，不需要初始化时一次性创建全部文件。
